Terminal Agent Tools Now Reach Any Visible Terminal

The send_to_terminal and get_terminal_output tools now work on foreground terminals — any terminal currently visible in the terminal panel, not only background terminals created by the agent. This means an agent can read output from a running REPL, send input to an interactive script, respond to a debugger prompt, or interact with any process already running in your workspace. Paired with the LLM-based prompt detection being removed (each terminal output chunk no longer triggers an extra LLM call), agent terminal interactions are now faster and cheaper in terms of token usage.

Key Takeaway: An agent can now interact with your Spring Boot REPL, your running Node.js dev server, or any open database CLI session — not just processes it spawned itself.
  1. LLM detection removed: Previously, every terminal output chunk triggered an LLM call to classify whether the terminal was waiting for input. That is gone. The agent now uses send_to_terminal directly and uses the question carousel to defer to you when needed.
  2. Progress messages: When the agent sends an answer to the terminal, the progress message shows which question is being answered (e.g., "Sending 'my-project' to terminal (replying to: What is your project name?)").
  3. Focus Terminal button: When the agent needs terminal input (password, npm init prompt), the question carousel includes a "Focus Terminal" button. Click it to take over and type directly. If you start typing, the carousel dismisses automatically and informs the agent you are handling it.
  4. Background notifications on by default: chat.tools.terminal.backgroundNotifications is now enabled by default — the agent receives instant notifications when background commands complete, time out, or require input.
Real Example: Your agent is running a Maven build in a background terminal. When the build finishes and triggers an interactive prompt for a test environment password, the agent is notified immediately, shows the question carousel, and either sends the password (if previously approved) or prompts you via the Focus Terminal button — no polling, no timeout.

#3

Enterprise: Group Policy to Control Agent Network Access

Admins can now define allowlists and denylists for which network domains agent tools (the fetch tool, integrated browser) can reach. Controlled via three group policy keys: ChatAgentNetworkFilter (enable/disable), ChatAgentAllowedNetworkDomains (wildcards supported, e.g., *.example.com), and ChatAgentDeniedNetworkDomains (takes precedence over allow). When the filter is enabled and both lists are empty, all external network access from agent tools is blocked. The rules also apply to the terminal sandbox when chat.agent.sandbox.enabled is active.

Key Takeaway: Enterprises with strict data egress requirements can now ensure agents cannot fetch from unauthorized external domains — critical for financial, healthcare, and government environments where data must stay within defined boundaries.
